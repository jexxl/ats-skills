# ATS 算法模型测试报告生成

## 目的

该 skill 会基于用户提供的完整源代码、完整输入输出数据和真实运行证据，生成 ATS 算法模型的“原始实验运行数据及测试报告”Markdown 与 Word 文档。该 skill 只负责文档生成，不负责整理 Docker 打包目录、构建镜像或导出镜像 tar 文件。

## 什么情况下调用？

- 用户说“帮我生成算法测试报告”“生成原始实验运行数据及测试报告”或类似表述
- 用户已有算法源码、输入数据、输出数据和运行记录，需要整理成 ATS 评审用 Word 报告
- 用户已有报告 Markdown，需要按本 skill 的模板和样式转换为 Word
- 用户要求检查 ATS 算法报告中的输入输出字段、测试数据、测试结果或占位符是否完整

如果用户要求“打包算法”“构建 Docker”“导出镜像 tar”或“提交云计算平台”，应使用打包 skill，而不是本 skill。

## 文档交付格式

术语统一：

- **算法编号**：业务/文档编号，采用 `课题编号-专题编号-基础(J)/专用(Z)-模型编号` 格式，例如 `4-4-Z-1`。Word 文档表格中的业务字段名仍保留为 **模型编号**，其值填写该算法编号。
- **报告 Markdown**：`{算法编号}_原始实验运行数据及测试报告.md`，是 Word 内容的唯一来源。
- **报告 Word**：`{算法编号}_原始实验运行数据及测试报告.docx`，由报告 Markdown 通过 `scripts/build_docs.py` 生成。

推荐交付目录：

```text
docs/
├── {算法编号}_原始实验运行数据及测试报告.md
└── {算法编号}_原始实验运行数据及测试报告.docx
```

## 工作流程

### Step 1: 收集基本信息

向用户一次性询问以下必须由用户提供或明确确认的信息（未提供的项目必须追问，不得跳过）。可从源码、数据样例或运行记录推断的信息，先自动推断再请用户确认。

1. **算法编号**：例如 `4-4-Z-10`，将写入文档中的模型编号字段
2. **算法名称**：用于文档标题和测试结论
3. **专题名称**
4. **模型功能描述**
5. **输入数据概括**
6. **输出数据概括**
7. **模型服务场景**
8. **上游接口模型编号**
9. **下游接口模型编号**
10. **交付时间**
11. **责任单位**
12. **完整源代码位置**：入口文件、模块、配置读取逻辑和依赖说明
13. **完整输入数据位置**：报告涉及的全部原始输入和配置文件
14. **完整输出数据位置**：真实运行产生的全部输出文件
15. **真实运行证据**：实际执行命令或流程、运行环境、日志摘要、成功/失败状态和结果核验依据

不得为专题名称、模型服务场景、上游接口模型编号、下游接口模型编号、交付时间、责任单位等业务字段设置默认值。即使值为 `-`，也必须来自用户提供或明确确认。

根据完整源代码、样例数据和运行记录自动推断以下信息，并在整理结果后请用户确认：算法模型简介、算法模型流程图、输入/输出文件与字段、可泛化配置参数、测试数据摘要、测试结果摘要、输入/输出概括。

### Step 2: 检查源码、数据和运行证据完整性

生成正式报告前必须确认：

- 源代码完整，能够说明算法入口、核心计算逻辑、依赖和配置读取方式
- 输入数据完整，覆盖本次测试实际使用的全部文件
- 输出数据完整，覆盖本次测试真实生成的全部结果文件
- 运行证据真实，包含执行方式、环境信息、日志摘要和结果核验方式
- 对 `npz`、模型权重、二进制矩阵等不可直接预览格式，必须提供可读摘要；必要时在输出目录中补充同名 `.txt` 预览文件

如果缺少完整源码、完整输入数据、完整输出数据或真实运行证据，不得生成最终报告；应先向用户追问缺失材料。

### Step 3: 准备测试报告 Markdown

使用 Step 1 收集和自动推断的信息，以及真实运行结果，填写报告 Markdown，不得留占位符。

**docs/{算法编号}_原始实验运行数据及测试报告.md**：参考 `template/docs/{算法编号}_原始实验运行数据及测试报告.md` 填写算法模型基本信息、输入数据要求、输出数据要求、算法模型简介、算法模型流程图、测试数据与配置、测试过程、测试结果、测试结论。

输入/输出字段必须严格使用模板格式，否则 Word 转换脚本会失败并提示修正：

- 每个输入或输出数据块必须先写一行数据名称，下一行写：`` (`文件名`) ``
- 紧跟一个 Markdown 表格，表头必须完全是：`字段中文 | 字段英文 | 字段类型`
- 字段行必须逐行填写，不得改成项目符号列表、JSON、逗号分隔文本、合并单元格或改表头名称
- 多个输入或输出数据按同一格式连续追加

算法模型简介必须聚焦算法层面，不得写成代码说明、工程说明或运行说明：

- 应描述算法要解决的问题、适用场景、核心思想、输入输出之间的数学关系、主要计算步骤、目标函数、约束条件、关键参数含义和结果解释
- 可使用必要公式说明算法原理，并在公式后解释变量含义
- 不得出现代码相关要素，包括源文件路径、模块名、类名、函数名、变量名、API 调用、第三方库、依赖包、配置文件路径、命令行、日志、异常处理、数据读写实现细节
- 不得描述“程序读取/调用/遍历/保存”等实现动作；应改写为“模型以……为输入”“算法计算……”“结果表示……”
- 不得引用 `main.py`、`src/`、`data/input/`、`data/output/`、`config.json`、`pyproject.toml` 等工程文件或目录

算法模型简介中的公式必须使用 pandoc 可稳定转换为 Word 的简单 LaTeX 数学语法：

- 行内公式使用 `$...$` 或 `\(...\)`；独立公式使用 `$$...$$` 或 `\[...\]`
- 允许使用上下标、分式、根号、求和、积分、极值、希腊字母、比较符、集合符号和 `\operatorname{...}`、`\mathrm{...}`、`\text{...}` 等常见命令
- 多行推导拆成多个独立公式块，不使用 LaTeX 对齐环境
- 禁止使用 `\newcommand`、`\def`、`\DeclareMathOperator`、`\label`、`\ref`、`\eqref`、`\tag`
- 禁止使用 `equation`、`align`、`aligned`、`gather`、`multline`、`array`、`matrix`、`pmatrix`、`bmatrix` 等 LaTeX 环境；如需矩阵或分段逻辑，优先改写为正文说明或 Markdown 表格

报告测试部分以模板为准，必须基于真实文件和真实运行结果填写：

- 保留模板中的固定章节和小节，不得把 `输出文件清单` 移到 `测试结果`
- `运行方式` 必须填写用户提供并实际执行的运行方式，不得替换为默认 Docker 命令
- 大型 CSV/Parquet/XML/JSON 不得整篇贴入 Word，只保留代表性节选或统计摘要
- `npz`、模型权重等不可直接预览格式必须写出可读摘要；必要时同时在输出目录中提供同名 `.txt` 预览文件
- 不得通过机械拼接其他文档或照抄某个算法样例来代替真实输入/输出展开

### Step 4: 生成测试报告 Word

确认 Step 3 的 Markdown 报告已补全后，生成同名 Word 文档。默认且仅生成这一份 Word。

Word 生成规则：

- `docx-reference/experiment-report-reference.docx` 是固定的 pandoc reference-doc，只用于继承页面、字体、标题和表格样式；其中正文不会进入最终报告
- Word 内容唯一来源是 `docs/{算法编号}_原始实验运行数据及测试报告.md`；不得直接编辑 reference docx 来补正文
- PlantUML 流程图由 `build_docs.py` 渲染并插入 Word；生成后脚本会统一处理分页、表格、字体和段落格式
- 文档构建依赖由 `scripts/build_docs.py` 的 PEP 723 声明提供，不写入算法项目自身依赖文件

Word 生成脚本：

```bash
uv run scripts/build_docs.py --docs-dir {报告docs目录} --algorithm-id {算法编号}
```

如果当前工作目录不是本 skill 目录，则将 `scripts/build_docs.py` 替换为本 skill 中该脚本的实际路径。`--docs-dir` 指向已填写完成的报告目录，脚本会在同一目录输出 `{算法编号}_原始实验运行数据及测试报告.docx`。

生成前必须确认：

- Markdown 中不存在 `{算法编号}`、`{算法名称}`、`{文件名}`、`{参数}`、`{责任单位}` 等未替换占位符
- 固定章节齐全：`算法模型基本信息`、`算法模型简介`、`算法模型流程图`、`测试数据与配置`、`测试过程`、`测试结果`、`测试结论`
- 固定小节齐全：`测试输入文件清单`、`实际输入数据`、`可泛化参数`、`运行方式`、`输出文件清单`
- `实际输入数据` 和 `测试结果` 均有真实数据表格、代码片段或二进制摘要，而不是模板示例、旧测试说明或预期值

### 完成

文档生成完成后向用户汇报：

- 最终 Markdown：`docs/{算法编号}_原始实验运行数据及测试报告.md`
- 最终 Word：`docs/{算法编号}_原始实验运行数据及测试报告.docx`
- 本次报告使用的源码、输入数据、输出数据和运行证据来源
- 由用户明确确认而非自动推断的业务字段
