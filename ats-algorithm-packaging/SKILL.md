# 云计算平台算法模型打包

## 目的

该skill会按照统一的目录结构和文件命名规范，将算法模型及其相关文档、输入输出数据等打包成一个标准化的格式，方便后续的部署和使用。

## 什么情况下调用？

- 用户说"帮我打包算法"、"把这个算法打包成 Docker"或类似表述
- 用户提到要将 Python 算法提交到云计算平台
- 用户的算法代码已经写好，需要整理目录结构并构建镜像
- 用户明确提到算法编号（如 `4-4-Z-1`）和打包需求

## 打包格式

术语统一：

- **算法编号**：业务/文档编号，采用 `课题编号-专题编号-基础(J)/专用(Z)-模型编号` 格式，例如 `4-4-Z-1`。Word 文档表格中的业务字段名仍保留为 **模型编号**，其值填写该算法编号。
- **打包编号**：文件系统和 Docker 使用的编号，由算法编号转换得到：前缀加 `algo`，并全部小写，例如 `4-4-Z-1` 转为 `algo4-4-z-1`。
- 目录名、Docker 镜像名、tar 归档名统一使用 **打包编号**；Docker 镜像标签严格使用 `v1`，不得使用 `latest`；文档文件名和文档内容中的模型编号统一使用 **算法编号**。

```text
{打包编号}/
├── data/
│   ├── input/          # 所有算法运行所需输入数据（包括配置文件）
│   └── output/         # 所有算法运行后生成的输出数据
├── docs/
│   ├── {算法编号}_原始实验运行数据及测试报告.md
│   └── {算法编号}_原始实验运行数据及测试报告.docx
├── src/                # 完整代码文件
├── Dockerfile          # 定义算法运行环境与依赖，保证结果可复现
├── .dockerignore
├── params.json         # docker run 额外参数，供云计算平台使用
├── pyproject.toml      # Python项目配置文件
├── uv.lock             # Python项目依赖锁定文件
└── {打包编号}.tar       # Docker 镜像文件，用于上传云计算平台
```

## 配置参数规范

**云计算平台不支持向容器传递命令行参数**，因此所有可泛化的算法配置参数（如阈值、时间范围、区域编码、模型路径等）必须通过配置文件传入，而不是 `argparse`、`sys.argv` 或任何命令行参数机制。

规范要求：

- 可变参数放入 `data/input/` 中的配置文件（如 `config.json`、`config.toml`、`config.yaml`）
- 算法代码在启动时从该配置文件读取参数，而非从命令行读取
- Dockerfile 的 `CMD` 只写入口命令，不附加任何参数：`["python", "main.py"]`
- 若用户代码中存在 `argparse` 或 `sys.argv`，必须在打包前将其重构为读取配置文件

## params.json 说明

`params.json` 用于向云计算平台传递额外的 `docker run` 启动参数。平台在运行容器时会读取该文件，并将其中的键值对转换为对应的 `docker run` 选项。

示例：

```json
{ "gpus": "all" }
```

等价于在 `docker run` 中添加 `--gpus all`，用于启用 GPU 支持。

**默认值**：若算法不需要特殊运行参数，`params.json` 内容为空对象 `{}`。打包时默认生成空对象，用户可在上传前根据云平台要求修改。

## 工作流程

### Step 1: 收集基本信息

向用户一次性询问以下必须由用户提供的信息（未提供的项目必须追问，不得跳过）。可从源码、数据样例或配置文件推断的信息，不在此处逐项询问，先自动推断再请用户确认。

1. **算法编号**：例如 `4-4-Z-10`，将写入文档中的模型编号字段；打包编号由该值自动推导，不单独询问
2. **源代码位置**：用户算法代码所在的目录或文件路径
3. **算法名称**：用于文档标题的中文名称
4. **专题名称**: 默认为 `多方式异智系统自主闭环跨域管控决策计算及算法`。仅在用户需要覆盖默认值时询问。
5. **算法服务场景**：必须从以下候选项中选择一个：
   - 城市路网群智协同信号控制
   - 城市路网异智车辆路径规划与信号控制跨域协同决策
   - 城市通勤走廊多模式交通自主协同运行
   - 面向轨道激增客流应急疏散的多方式自主协同管控与服务
6. **接口模型编号与交付信息**：上游接口模型编号和下游接口模型编号默认为 `-`；交付时间默认为当前日期；责任单位默认为同济大学。仅在用户需要覆盖默认值时询问。
7. **特殊运行要求**：如 Python 版本不是 3.11、需要 GPU、需要系统库、依赖版本必须固定等

根据源代码和样例数据自动推断以下信息，并在整理结果后请用户确认：算法功能描述、算法模型简介、算法模型流程图、入口文件、Python 依赖、输入/输出文件与字段、可泛化配置参数、测试数据摘要、预期输出摘要、输入/输出概括。

### Step 2: 创建目录结构

将算法编号转换为打包编号，并以打包编号为根目录名创建以下目录结构。转换规则：`{算法编号}` 前加 `algo`，并全部小写，例如 `4-4-Z-10` 转为 `algo4-4-z-10`。

```
{打包编号}/
├── data/
│   ├── input/
│   └── output/
├── docs/
└── src/
```

### Step 3: 整理源代码

将用户的算法代码复制或移动到对应目录：

- 入口文件（`main.py`）放在根目录（与 `src/` 同级）
- 其余 `.py` 模块和子包放在 `src/`
- 不包含 `.venv/`、`__pycache__/`、`*.pyc`

**重要：检查命令行参数用法。** 复制代码后，检查所有 `.py` 文件中是否存在 `argparse`、`sys.argv`、`click` 等命令行参数解析逻辑。若存在，必须将其重构为从配置文件读取参数，原因见"配置参数规范"一节。重构方式：

1. 将所有命令行参数的默认值整理为配置文件（如 `data/input/config.toml`）
2. 在代码中改为读取该配置文件（`json.load` / `tomllib.load` / `yaml.safe_load`）
3. 告知用户修改了哪些代码，以及配置文件的字段含义

**重要：检查输出可预览性。** 若算法主要输出 `npz`、模型权重、二进制矩阵等不可直接预览格式，必须在打包时补充可读预览输出（同名的 `.txt` 摘要文件），并写入 `data/output/`，供 Step 9 验证和 Step 10 报告填写使用。`.txt` 摘要文件只作为核心输出的辅助预览，不作为独立算法结果。

### Step 4: 整理输入示例数据

将输入数据复制到 `data/input/`。若算法存在可泛化参数，需在 `data/input/` 中创建对应的配置文件（如 `config.toml`），并填入合理的默认值作为示例。云计算平台用户通过替换该文件来调整算法行为。

### Step 5: 生成 pyproject.toml

参考 `template/pyproject.toml` 模板，根据收集到的信息填写：

- `name` 填写打包编号
- `requires-python` 根据实际 Python 版本填写
- `dependencies` 填写所有第三方依赖及版本约束
- 镜像源保留阿里云配置，以保证在中国网络环境下正常安装

### Step 6: 运行 uv lock

在算法根目录下运行以下命令，生成 `uv.lock` 锁定文件：

```bash
cd {打包编号}
uv lock
```

如果运行失败，检查 `pyproject.toml` 中的依赖描述是否有误，并根据错误信息修正。

### Step 7: 生成 Dockerfile 和 .dockerignore

参考 `template/Dockerfile` 模板生成 Dockerfile。决策点：

- **是否需要系统库**：如果依赖 `opencv`、`libsumo`、matplotlib 中文字体等需要系统级库的包，使用 `apt-get install` 安装；若只有纯 Python 依赖则无需此步骤
- **Python 版本**：与 `pyproject.toml` 中的 `requires-python` 保持一致
- **入口命令**：`CMD` 只填写 `["python", "main.py"]` 或用户指定的入口，**不得附加任何命令行参数**。云计算平台不支持向容器传递命令行参数，所有参数须通过 `data/input/` 中的配置文件传入

同时在根目录生成 `params.json`（默认内容为空对象，如需 GPU 等特殊参数由用户在上传前修改）：

```json
{}
```

以及 `.dockerignore`（参考 `template/.dockerignore`，并根据实际需要补充）。

### Step 8: 构建 Docker 镜像

在算法根目录下运行：

```bash
cd {打包编号}
docker build -t {打包编号}:v1 .
```

如果构建失败，根据错误信息排查：依赖安装失败检查包名和版本约束；系统库缺失在 Dockerfile 中增加 `apt-get install` 步骤；代码路径错误检查 `COPY` 指令。

### Step 9: 验证镜像

构建成功后，运行镜像验证：

```bash
# Linux/macOS
docker run --rm \
  -v $(pwd)/data/input:/app/data/input \
  -v $(pwd)/data/output:/app/data/output \
  {打包编号}:v1
```

```powershell
# Windows PowerShell
docker run --rm `
  -v "${PWD}/data/input:/app/data/input" `
  -v "${PWD}/data/output:/app/data/output" `
  {打包编号}:v1
```

验证内容：

- 镜像启动无报错，算法能读取输入数据
- 输出文件正确生成，且所有输出均可预览；对 `npz` 等不可直接预览格式，需同时检查对应文本预览文件已生成且内容可读
- 至少执行两次对照验证：一次使用默认配置，一次修改可泛化参数；两次运行的输出结果应存在符合参数含义的差异

### Step 10: 准备测试报告 Markdown

镜像验证完成后，使用 Step 1 收集和自动推断的信息，以及 Step 9 的真实运行结果，填写报告 Markdown，不得再重复询问用户，也不得留占位符。

**docs/{算法编号}_原始实验运行数据及测试报告.md**：参考 `template/docs/{算法编号}_原始实验运行数据及测试报告.md` 填写算法模型基本信息、输入数据要求、输出数据要求、算法模型简介、算法模型流程图、测试数据与配置、测试过程、测试结果、测试结论。

输入/输出字段必须严格使用模板格式，否则 Word 转换脚本会失败并提示修正：

- 每个输入或输出数据块必须先写一行数据名称，下一行写：`` (`文件名`) ``
- 紧跟一个 Markdown 表格，表头必须完全是：`字段中文 | 字段英文 | 字段类型`
- 字段行必须逐行填写，不得改成项目符号列表、JSON、逗号分隔文本、合并单元格或改表头名称
- 多个输入或输出数据按同一格式连续追加

算法模型简介必须聚焦算法层面，不得写成代码说明、工程说明或运行说明：

- 应描述算法要解决的问题、适用场景、核心思想、输入输出之间的数学关系、主要计算步骤、目标函数、约束条件、关键参数含义和结果解释
- 可使用必要公式说明算法原理，并在公式后解释变量含义
- 不得出现代码相关要素，包括源文件路径、模块名、类名、函数名、变量名、API 调用、第三方库、依赖包、配置文件路径、命令行、Docker、日志、异常处理、数据读写实现细节
- 不得描述“程序读取/调用/遍历/保存”等实现动作；应改写为“模型以……为输入”“算法计算……”“结果表示……”
- 不得引用 `main.py`、`src/`、`data/input/`、`data/output/`、`config.json`、`pyproject.toml` 等工程文件或目录

算法模型简介中的公式必须使用 pandoc 可稳定转换为 Word 的简单 LaTeX 数学语法：

- 行内公式使用 `$...$` 或 `\(...\)`；独立公式使用 `$$...$$` 或 `\[...\]`
- 允许使用上下标、分式、根号、求和、积分、极值、希腊字母、比较符、集合符号和 `\operatorname{...}`、`\mathrm{...}`、`\text{...}` 等常见命令
- 多行推导拆成多个独立公式块，不使用 LaTeX 对齐环境
- 禁止使用 `\newcommand`、`\def`、`\DeclareMathOperator`、`\label`、`\ref`、`\eqref`、`\tag`
- 禁止使用 `equation`、`align`、`aligned`、`gather`、`multline`、`array`、`matrix`、`pmatrix`、`bmatrix` 等 LaTeX 环境；如需矩阵或分段逻辑，优先改写为正文说明或 Markdown 表格

算法模型流程图必须用 PlantUML 表达算法层面的计算链路，不得写成简化运行流程或文件处理流程：

- 流程图应覆盖核心输入形成、关键中间状态、主要计算步骤、必要判断分支、输出结果形成，原则上不少于 8 个实质计算步骤；特别简单的算法也不得少于 6 个实质步骤
- 至少包含一个与算法逻辑有关的条件判断或分支，例如输入质量、约束可行性、候选有效性、决策规则、状态边界、异常观测处理等；确无分支时，应在正文简介中说明原因
- 节点文本应使用通用算法语义，例如“形成状态表示”“计算目标函数”“更新中间状态”“生成输出指标”，不得使用“读取文件”“加载配置”“保存结果”等工程动作作为主体
- 流程图内容应与算法模型简介中的问题、公式、关键参数和输出解释一致，不得只用 4-5 个泛化节点概括为“输入-计算-输出”
- PlantUML 使用简单 `start`、动作节点、`if/else/endif`、`stop` 等结构，避免复杂样式、泳道、外部图片或转换脚本不稳定支持的语法

报告测试部分以模板为准，必须基于真实文件和真实运行结果填写：

- 保留模板中的固定章节和小节，不得把 `输出文件清单` 移到 `测试结果`
- 大型 CSV/Parquet/XML/JSON 不得整篇贴入 Word，只保留代表性节选或统计摘要
- `npz`、模型权重等不可直接预览格式必须写出可读摘要；必要时同时在 `data/output/` 中提供同名 `.txt` 预览文件
- `.txt` 可读摘要文件不得在 `测试结果` 中设置独立的“可读摘要文件”小节，也不得作为独立结果重复展开；应在 `输出文件清单` 中说明其辅助预览用途，并在对应核心输出小节中直接展示解析后的表格、统计摘要或代表性节选
- 不得通过机械拼接其他文档或照抄某个算法样例来代替真实输入/输出展开

### Step 11: 生成测试报告 Word

确认 Step 10 的 Markdown 报告已补全后，生成同名 Word 文档。默认且仅生成这一份 Word。

Word 生成规则：

- `docx-reference/experiment-report-reference.docx` 是固定的 pandoc reference-doc，只用于继承页面、字体、标题和表格样式；其中正文不会进入最终报告。
- Word 内容唯一来源是 `docs/{算法编号}_原始实验运行数据及测试报告.md`；不得直接编辑 reference docx 来补正文。
- PlantUML 流程图由 `build_docs.py` 渲染并插入 Word；生成后脚本会统一处理分页、表格、字体和段落格式。
- 文档构建依赖由 `scripts/build_docs.py` 的 PEP 723 声明提供，不写入算法项目自身 `pyproject.toml`。

Word 生成脚本：

```bash
uv run scripts/build_docs.py --docs-dir {打包编号}/docs --algorithm-id {算法编号}
```

如果当前工作目录不是本 skill 目录，则将 `scripts/build_docs.py` 替换为本 skill 中该脚本的实际路径。`--docs-dir` 指向已填写完成的交付目录 `docs/`，脚本会在同一目录输出 `{算法编号}_原始实验运行数据及测试报告.docx`。

生成前必须确认：

- Markdown 中不存在 `{算法编号}`、`{算法名称}`、`{文件名}`、`{参数}` 等未替换占位符
- 固定章节齐全：`算法模型基本信息`、`算法模型简介`、`算法模型流程图`、`测试数据与配置`、`测试过程`、`测试结果`、`测试结论`
- 固定小节齐全：`测试输入文件清单`、`实际输入数据`、`可泛化参数`、`运行方式`、`输出文件清单`
- `算法模型流程图` 使用 PlantUML，包含足够的算法计算步骤和必要判断分支，不得是过短的工程运行流程
- `实际输入数据` 和 `测试结果` 均有真实数据表格、代码片段或二进制摘要，而不是模板示例、旧测试说明或预期值

### Step 12: 将 Docker 镜像导出为 tar 文件

在算法根目录下运行以下命令，将 Docker 镜像导出为 tar 文件：

```bash
docker save -o {打包编号}.tar {打包编号}:v1
```

### 完成

打包完成后向用户汇报：

- 生成的目录结构
- Docker 镜像名称和标签：`{打包编号}:v1`
- 最终交付文件：`{打包编号}.tar`（Docker 镜像，位于算法根目录）
- 最终文档：`docs/{算法编号}_原始实验运行数据及测试报告.docx`
- 提示用户 `params.json` 默认为 `{}`，如需 GPU 支持则填写 `{"gpus":"all"}`，其他 `docker run` 参数同理
